RESEARCHONE ROUTE A IMPLEMENTATION SPEC FOR CODING AGENT

Purpose
Create the next working version of ResearchOne so it actually fulfills the Route A handoff and current deployment intent:
- frontend/UI stays on Vercel
- backend/orchestrator stays on Emma runtime VM
- Postgres stays on Emma postgres VM
- Redis stays on Emma redis VM
- OpenRouter remains the server-side model layer
- system must support disciplined autonomous research discovery, ingestion, retrieval, contradiction preservation, and report generation
- system must not become a generic hallucination chatbot

This document is implementation-first. Do not redesign the whole product. Patch the current repo into the correct shape.

Repository baseline
Current repo already has:
- Express backend
- BullMQ queues/workers
- Postgres schema with corpus/report tables
- OpenRouter role orchestration
- hybrid retrieval (vector + FTS)
- Atlas export pipeline
- Vite frontend

Confirmed current gaps to fix
1. Split deployment gap
   - frontend assumes same-origin /api and same-origin socket.io
   - that works only when frontend is served from runtime VM
   - it does not cleanly support Vercel frontend + Emma backend

2. PDF ingestion gap
   - file upload route accepts PDFs
   - ingestion service only handles web_url and raw text
   - PDF uploads do not actually become text/chunks/embeddings

3. Markdown and richer text ingestion gap
   - no first-class handling for markdown/plain structured text imports beyond raw text field

4. Research discovery gap
   - research pipeline retrieves only from existing corpus
   - it does not perform self-led external discovery when corpus is sparse or insufficient
   - user explicitly wants normal research behavior, not only PDF/MD uploads

5. Epistemic persistence gap
   - claims, contradictions, and report citations exist in schema
   - current pipeline does not materially persist extracted claims, explicit contradictions, or section-level citations after report generation

6. Atlas export path mismatch
   - app writes exports to process cwd exports directory
   - runtime nginx script points /exports at /opt/researchone/backend/exports/
   - these paths must be unified

7. Redis auth readiness gap
   - setup script can enable requirepass
   - backend Redis client does not support password env/config

8. Fallback coverage gap
   - only some OpenRouter roles have fallbacks
   - retriever, skeptic, verifier do not

Non-negotiables
- Keep the UI on Vercel
- Do not move the UI to the laptop
- Do not self-host giant frontier models on the three current Emma VMs
- Keep OpenRouter on the server side only
- Preserve the research philosophy:
  - dense centers are context, not truth
  - outliers are leads, not verdicts
  - bridges are high-value
  - contradictions are first-class
  - every major claim must preserve epistemic tier distinctions
- Do not flatten this into generic RAG boilerplate
- Do not store OpenRouter keys or backend secrets in Vercel client env

Target deployment topology
1. Vercel project
   - serves only the frontend build
   - frontend uses configurable backend URLs
   - no model calls from browser

2. Emma runtime VM
   - Express API
   - Socket.IO
   - BullMQ workers
   - search/discovery orchestration
   - ingestion/fetch/chunk/embed/report pipeline
   - exports directory served via backend and/or nginx

3. Emma Postgres VM
   - corpus store
   - pgvector
   - claims/contradictions/reports/report citations

4. Emma Redis VM
   - queues/job state/cache

Implementation goals
A. Make split deployment real
B. Make ingestion actually support PDF, markdown, txt, html, URL discovery
C. Add autonomous external research discovery controlled by the planner/reasoner loop
D. Persist claims, contradictions, citations, and discovery provenance
E. Keep Atlas useful for investigation, not just pretty export files
F. Update README/env/deployment docs to match the real deployment path

SECTION 3 - AUTONOMOUS EXTERNAL RESEARCH DISCOVERY

Objective
When the user starts a research run, the system should not only search the existing corpus. It should be able to decide that more evidence is needed, perform external discovery, ingest high-value sources, and then continue disciplined retrieval/reasoning.

Important design rule
This is not open-ended autonomous browsing chaos.
The model may propose discovery targets, but the backend executes bounded, auditable search/discovery actions.

Required pipeline change
Insert a discovery stage between planning and internal retrieval.
New research flow:
1. planner creates plan
2. discovery planner decides whether external discovery is needed
3. external discovery adapters fetch candidate sources/snippets/metadata
4. source selector ranks which candidates to ingest
5. ingestion jobs bring selected sources into corpus
6. embeddings complete for new chunks
7. retrieval runs across enriched corpus
8. retriever/reasoner/skeptic/synthesizer/verifier continue
9. claims/contradictions/citations are persisted
10. report saved

Required new services
1. backend/src/services/discovery/discoveryOrchestrator.ts
   Responsibilities:
   - accept runId, plan, query, optional tags
   - ask model whether discovery is needed
   - execute bounded search adapters
   - merge/dedupe candidate sources
   - score/rank candidates
   - enqueue ingestion for selected sources
   - wait for ingestion/embedding completion before continuing (bounded timeout)

2. backend/src/services/discovery/providerTypes.ts
   Define normalized types:
   - SearchQuery
   - SearchResultCandidate
   - DiscoverySource
   - DiscoveryRunSummary

3. backend/src/services/discovery/providers/
   Add provider abstraction.
   Initial implementation must support at least:
   - generic web search provider interface
   - URL fetch provider (already partly present through URL ingestion)
   Optional adapters can be added later for scholarly or structured corpora.

Important implementation note
Do not hardwire business logic to any single provider. Use a provider interface and env-configured provider selection.

Discovery behavior requirements
- bounded result count
- dedupe by normalized URL/content hash
- log which search queries were executed
- log why a candidate was ingested or skipped
- never treat search ranking as truth ranking
- preserve candidate metadata even when skipped if useful for audit

New role/prompt behavior
Add a discovery-planning sub-step in the planner or create a dedicated discovery role.
The discovery planner should output structured JSON like:
- need_external_discovery: boolean
- rationale
- discovery_queries[]
- target_source_types[]
- preferred evidence tiers
- max_sources_to_ingest
- exclusion patterns
- what would count as disconfirming evidence

Search provider env strategy
Add backend env variables such as:
- SEARCH_PROVIDER=generic
- SEARCH_PROVIDER_API_KEY=
- SEARCH_PROVIDER_BASE_URL=
- MAX_EXTERNAL_DISCOVERY_RESULTS=25
- MAX_EXTERNAL_INGEST_PER_RUN=10
- DISCOVERY_ENABLED=true

Do not block implementation on one specific commercial provider.
Build the abstraction so one adapter can be used now and others later.

Acceptance criteria
- a research run with sparse corpus can trigger external discovery
- discovered URLs are auditable per run
- selected discovery results are ingested automatically
- retrieval after discovery uses newly ingested material
- report metadata shows discovery happened and which sources were pulled in

SECTION 5 - PERSIST CLAIMS, CONTRADICTIONS, AND REPORT CITATIONS

Objective
The report pipeline must leave behind machine-queryable epistemic artifacts, not just prose.

Required orchestration changes
Modify backend/src/services/reasoning/researchOrchestrator.ts

After synthesizer/verifier but before final report completion:
1. run claim extraction step
2. run contradiction extraction step
3. run citation mapping step
4. persist all outputs inside same report save phase or coordinated transaction boundaries

Add new services
1. backend/src/services/reasoning/claimExtractor.ts
   Input:
   - query
   - retrieved chunks
   - reasoner output
   - synthesizer output
   Output:
   - normalized claim records
   Each claim record should contain:
   - claim_text
   - evidence_tier
   - confidence
   - supporting_chunk_ids
   - source_ids
   - tags
   - whether the claim is conclusion-critical

2. backend/src/services/reasoning/contradictionExtractor.ts
   Input:
   - retrieved chunks
   - claim set
   - skeptic output
   Output:
   - contradiction pairs with descriptions and severity

3. backend/src/services/reasoning/citationMapper.ts
   Input:
   - parsed report sections
   - retrieved chunks
   - extracted claims
   Output:
   - section-level citations linked to chunks/sources/claims

Implementation rules
- do not hallucinate citations
- if no direct citation mapping confidence is high enough, leave section partially uncited and note it in verifier metadata
- preserve uncertainty explicitly

Acceptance criteria
- new finalized report creates rows in claims, contradictions, and report_citations
- corpus views for contradictions and claims are meaningfully populated
- reports can be inspected later without reparsing prose
